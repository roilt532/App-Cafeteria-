1. Introducción

En este documento se define la API REST que servirá como interfaz entre el frontend hecho en React y el backend con Django REST Framework. La idea es que los dos equipos puedan trabajar a la vez sabiendo exactamente qué datos se envían y se reciben en cada petición.

Algunas cosas generales que aplican a toda la API:

  • URL base: https://api.piobite.es/api/v1/
  • Formato: Todas las peticiones y respuestas van en JSON (Content-Type: application/json).
  • Autenticación: Se usan tokens JWT. El token se manda en cada petición con la cabecera: Authorization: Bearer <token>

Hay dos roles de usuario:

RolQuién esQué puede hacer
ClienteAlumnos y personal del institutoVer productos, hacer pedidos, ver su historial
Dueño / StaffPersonal de la cafeteríaTodo lo anterior + gestionar inventario, pedidos y ver estadísticas

Las respuestas que devuelven listas usan paginación con este formato:

{
    "count": 48,
    "next": "https://api.piobite.es/api/v1/productos/?page=2",
    "previous": null,
    "results": [ ... ]
}

3. Flujo de Autenticación con Google SSO

Cuando el usuario pulsa "Iniciar sesión con Google" pasan los siguientes pasos:

  1. El frontend redirige a Google. Se le envía al usuario a la pantalla de login de Google con nuestro client_id y los permisos que necesitamos (email y perfil).
  2. El usuario elige su cuenta de Google y acepta los permisos.
  3. Google redirige de vuelta al frontend con un parámetro code en la URL. Este código es temporal y de un solo uso.
  4. El frontend envía ese código al backend. 
    POST /api/v1/auth/google/
{ "code": "4/0AX4XfWh..." }
  5. El backend usa ese código para hablar con Google. Le envía el código junto con nuestro client_secret al endpoint de Google https://oauth2.googleapis.com/token y recibe a cambio un id_token con los datos del usuario.
  6. El backend crea o recupera al usuario. Saca el email, nombre y foto del id_token. Si el usuario no existía en nuestra base de datos, lo crea con rol "cliente". Si ya existía, simplemente lo recupera.
  7. El backend genera tokens JWT propios y los devuelve al frontend:
    • access_token: dura 15 minutos, se usa para autenticar las peticiones.
    • refresh_token: dura 7 días, se usa para renovar el access cuando caduca.
    {
    "access": "eyJhbGciOiJIUzI1NiJ9...",
    "refresh": "eyJhbGciOiJIUzI1NiJ9...",
    "user": { "id": 1, "email": "alumno@piobaroja.es", "rol": "cliente" }
}
  8. El frontend guarda los tokens y redirige al usuario al catálogo. A partir de aquí, cada petición al backend incluye la cabecera Authorization: Bearer <access_token>.

Cuando el access token caduca (al cabo de 15 min), el frontend detecta que recibe un error 401, envía automáticamente el refresh token a POST /auth/refresh/ para conseguir un nuevo access token, y reintenta la petición original sin que el usuario se entere.

4. Flujo de un Pedido Completo

Este es el recorrido completo de un pedido desde que el cliente elige productos hasta que los recoge:

  1. El cliente mira el catálogo y añade productos al carrito.
    El frontend pide los productos al backend con GET /productos/ y los muestra. El carrito se gestiona en local (estado de React), no se envía nada al servidor todavía.
  2. El cliente elige la franja horaria de recogida.
    El frontend consulta las franjas disponibles con GET /franjas-horarias/?fecha=2025-04-15 y el usuario elige una.
  3. El cliente confirma el pedido.
    El frontend envía el carrito y la franja al backend: 
    POST /api/v1/pedidos/
{
    "items": [
        { "producto_id": 1, "cantidad": 1 },
        { "producto_id": 7, "cantidad": 2 }
    ],
    "franja_horaria_id": 3
}
    El backend comprueba el stock, calcula el total, genera un código de recogida (ej: PB2401) y crea el pedido con estado pendiente_pago.
  4. Se inicia el pago con Redsys.
    El frontend pide los datos de pago al backend con POST /pagos/iniciar/. El backend genera los parámetros firmados que Redsys necesita (importe en céntimos, número de pedido, firma HMAC SHA256) y se los devuelve al frontend.
  5. El usuario paga en Redsys.
    El frontend crea un formulario oculto con los datos de Redsys y lo envía automáticamente, redirigiendo al usuario a la pasarela. Ahí mete los datos de su tarjeta y confirma (puede pasar por verificación 3D Secure).
  6. Redsys avisa al backend del resultado.
    Redsys hace una llamada POST directa al backend (POST /pagos/notificacion/) de servidor a servidor. El backend verifica la firma, comprueba que el código de respuesta sea correcto (0000-0099 = pago OK) y actualiza el pedido a estado pendiente si el pago fue bien, o a pago_fallido si no.
  7. El usuario vuelve a la app.
    Redsys redirige al usuario de vuelta a nuestra web. El frontend le muestra la pantalla de confirmación con el código QR y el código de recogida.
  8. La cafetería prepara el pedido.
    El personal ve el pedido en su panel (GET /pedidos/admin/) y va cambiando su estado según avanza: 
    pendiente_pago → pendiente → preparando → listo → entregado
    Cada cambio se hace con: PATCH /pedidos/{id}/estado/
  9. El cliente recoge su pedido.
    El cliente enseña su código QR o dice su código (PB2401). El personal lo verifica con GET /pedidos/verificar/PB2401/, le entrega la comida y marca el pedido como entregado.

5. Gestión de Errores

Estructura de las respuestas de error

Todas las respuestas de error van a tener este formato para que el frontend siempre sepa cómo leerlas:

{
    "error": {
        "codigo": "TOKEN_EXPIRADO",
        "mensaje": "El token de acceso ha expirado.",
        "detalles": null
    }
}

En los errores de validación, el campo "detalles" incluye qué campo está mal:

{
    "error": {
        "codigo": "VALIDACION_FALLIDA",
        "mensaje": "Los datos enviados contienen errores.",
        "detalles": {
            "items": ["Este campo es obligatorio."],
            "franja_horaria_id": ["La franja seleccionada ya no está disponible."]
        }
    }
}

Códigos de estado HTTP utilizados

CódigoCuándo se usaEjemplo
200Todo ha ido bien—
201Se ha creado un recurso nuevo (pedido, producto...)—

400Datos de entrada incorrectos o falta algún campo"Los datos enviados contienen errores."

401El token ha caducado"El token de acceso ha expirado."
401El código de Google no es válido"No se pudo verificar la identidad con Google."

404No se encuentra el recurso (producto, pedido, código de recogida)"El pedido con código 'PB9999' no existe."
409Transición de estado no válida en un pedido"No se puede pasar de 'entregado' a 'preparando'."
422El pago ha sido rechazado por Redsys"El pago ha sido rechazado. Código Redsys: 0190."

429Demasiadas peticiones seguidas"Has excedido el límite de peticiones. Espera 60 segundos."
500Error interno del servidor"Ha ocurrido un error interno."
503La pasarela Redsys no está disponible"La pasarela de pago no está disponible ahora mismo."

Qué hace el frontend con cada error

CódigoQué hace el frontend
400Muestra los mensajes de error debajo de los campos del formulario que estén mal.
401 (token expirado)Intenta renovar el token automáticamente con POST /auth/refresh/. Si falla, manda al usuario al login.
401 (otros)Redirige al usuario a la pantalla de login.
403Muestra "No tienes permisos" y redirige a la página principal.
404Muestra pantalla de "No encontrado".
422 (pago)Muestra mensaje de error del pago y da la opción de intentarlo otra vez.
429Muestra aviso de que espere un poco antes de volver a intentarlo.
500 / 503Muestra pantalla de error genérica con botón de reintentar.
