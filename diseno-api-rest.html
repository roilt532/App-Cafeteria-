1. Introducción y Convenciones Generales

Este documento define la API REST que actúa como interfaz de comunicación entre el frontend (React) y el backend (Django REST Framework) de la aplicación de cafetería del Instituto Pío Baroja.

URL base

https://api.piobite.es/api/v1/

Formato de datos

Todas las peticiones y respuestas utilizan Content-Type: application/json.

Autenticación

La autenticación se realiza mediante tokens JWT (JSON Web Token). El token se envía en la cabecera de cada petición autenticada:

Authorization: Bearer <access_token>

Roles de usuario

RolDescripciónPermisos
clienteAlumno o personal del institutoVer catálogo, hacer pedidos, ver historial propio
dueño/staffPropietario o empleado de la cafeteríaTodo lo anterior + gestión de inventario, pedidos y estadísticas

Paginación

Los listados que devuelven múltiples recursos utilizan paginación por cursor:

{
  "count": 48,
  "next": "https://api.piobite.es/api/v1/productos/?page=2",
  "previous": null,
  "results": [ ... ]
}

2. Listado de Endpoints

2.1. Autenticación (Google SSO)

Método Ruta Descripción Auth Parámetros

POST /auth/logout/ Invalida el refresh token actual. Cualquier rol Body: { "refresh": "string" }
GET /auth/me/ Devuelve los datos del usuario autenticado. Cualquier rol —

2.3. Carrito

Nota: El carrito se gestiona en el frontend (almacenamiento local en el navegador). No existe un endpoint de carrito en el backend. El carrito se envía al backend solo cuando el usuario confirma el pedido, como parte del cuerpo de la petición POST /pedidos/.

2.5. Pagos (Redsys)

Método Ruta Descripción Auth Parámetros
POST /pagos/iniciar/ Genera los parámetros necesarios para redirigir al usuario a la pasarela Redsys. Cliente Body: { "pedido_id": 42 }
POST /pagos/notificacion/ Webhook que recibe la notificación de Redsys con el resultado del pago (URL de notificación). Redsys (IP verificada) Body: Ds_SignatureVersion, Ds_MerchantParameters, Ds_Signature
GET /pagos/resultado/ Página de retorno tras el pago. Redirige al frontend con el estado. Ninguna Query: Ds_SignatureVersion, Ds_MerchantParameters, Ds_Signature

Ejemplo de respuesta — POST /pagos/iniciar/

{
  "redsys_url": "https://sis-t.redsys.es:25443/sis/realizarPago",
  "Ds_SignatureVersion": "HMAC_SHA256_V1",
  "Ds_MerchantParameters": "eyJEc19NZXJjaGFudF9Nb...",
  "Ds_Signature": "qWgR7/bKpP0n9M1dGK3..."
}
Importante: El frontend recibe estos parámetros y crea un formulario oculto que envía al usuario a la URL de Redsys mediante POST. No se realiza la llamada por AJAX sino por redirección del navegador.

3. Flujo de Autenticación con Google SSO

A continuación se describe paso a paso el flujo completo de autenticación desde que el usuario pulsa "Iniciar sesión con Google" hasta que puede operar con la aplicación.

1
El usuario pulsa "Iniciar sesión con Google"
El frontend redirige al usuario a la página de consentimiento de Google OAuth 2.0, incluyendo el client_id, redirect_uri y los scopes necesarios (openid, email, profile).

https://accounts.google.com/o/oauth2/v2/auth?
  client_id=GOOGLE_CLIENT_ID&
  redirect_uri=https://piobite.es/auth/callback&
  response_type=code&
  scope=openid email profile&
  state=random_csrf_token
2
Google muestra la pantalla de consentimiento
El usuario selecciona su cuenta de Google (por ejemplo, su cuenta del instituto @piobaroja.es) y acepta los permisos solicitados.
3
Google redirige al frontend con un código de autorización
Google redirige al usuario de vuelta a la redirect_uri del frontend con un parámetro code en la URL.

https://piobite.es/auth/callback?code=4/0AX4XfWh...&state=random_csrf_token
4
El frontend envía el código al backend
El frontend extrae el code de la URL y lo envía al backend mediante una petición POST.

// Frontend → Backend
POST /api/v1/auth/google/
{ "code": "4/0AX4XfWh..." }
5
El backend intercambia el código con Google
El backend utiliza el code junto con su client_secret para solicitar a Google un id_token y un access_token a través del endpoint https://oauth2.googleapis.com/token.

// Backend → Google
POST https://oauth2.googleapis.com/token
{ client_id, client_secret, code, grant_type: "authorization_code", redirect_uri }
6
El backend valida los datos y crea/recupera al usuario
El backend decodifica el id_token de Google para obtener el email, nombre y avatar del usuario. Busca al usuario en la base de datos. Si no existe, lo crea con rol cliente por defecto. Si ya existe, recupera sus datos.
7
El backend genera tokens JWT propios y los devuelve al frontend
El backend genera un access_token (corta duración, 15 minutos) y un refresh_token (larga duración, 7 días) firmados con la clave secreta del servidor. Se devuelven al frontend junto con los datos del usuario.

// Backend → Frontend
{
  "access": "eyJhbGciOiJIUzI1NiJ9...",
  "refresh": "eyJhbGciOiJIUzI1NiJ9...",
  "user": { "id": 1, "email": "alumno@piobaroja.es", "rol": "cliente" }
}
8
El frontend almacena los tokens y redirige al usuario
El frontend almacena el access_token en memoria (variable de estado) y el refresh_token en una cookie httpOnly o en localStorage. A partir de este momento, todas las peticiones al backend incluyen la cabecera Authorization: Bearer <access_token>. El usuario es redirigido al catálogo de productos.
Renovación automática: Cuando el access_token expira, el frontend detecta una respuesta 401, envía automáticamente el refresh_token a POST /auth/refresh/ para obtener un nuevo access_token, y reintenta la petición original de forma transparente para el usuario.

4. Flujo de un Pedido Completo

Ciclo de vida completo de un pedido desde la selección de productos hasta la recogida en la cafetería.

1
El cliente explora el catálogo y añade productos al carrito
El frontend solicita los productos al backend (GET /productos/) y los muestra al usuario. El carrito se gestiona localmente en el navegador (estado de React). No se envía nada al backend en este paso.
2
El cliente selecciona la franja horaria de recogida
El frontend solicita las franjas disponibles (GET /franjas-horarias/?fecha=2025-04-15). El usuario elige una franja entre las que tienen disponibilidad.
3
El cliente confirma el pedido
El frontend envía el carrito y la franja seleccionada al backend.

// Frontend → Backend
POST /api/v1/pedidos/
{
  "items": [
    { "producto_id": 1, "cantidad": 1 },
    { "producto_id": 7, "cantidad": 2 }
  ],
  "franja_horaria_id": 3
}
El backend valida la disponibilidad de stock, calcula el total, genera un código único de recogida (ej: PB2401) y crea el pedido con estado pendiente_pago. Devuelve el ID del pedido.
4
El frontend inicia el proceso de pago con Redsys
El frontend solicita al backend los datos necesarios para redirigir al usuario a la pasarela de pago.

// Frontend → Backend
POST /api/v1/pagos/iniciar/
{ "pedido_id": 42 }

// Backend → Frontend
{
  "redsys_url": "https://sis-t.redsys.es:25443/sis/realizarPago",
  "Ds_SignatureVersion": "HMAC_SHA256_V1",
  "Ds_MerchantParameters": "eyJEc19NZXJjaGFudF9Nb...",
  "Ds_Signature": "qWgR7/bKpP0n9M1dGK3..."
}
El backend genera los Ds_MerchantParameters (codificados en Base64) que contienen: número de pedido, importe en céntimos, moneda (978 = EUR), URL de notificación y URL de retorno. Firma los datos con la clave secreta del comercio para generar Ds_Signature.
5
El usuario es redirigido a Redsys y realiza el pago
El frontend crea un formulario oculto con los parámetros de Redsys y lo envía automáticamente (POST). El usuario ve la página de pago de Redsys donde introduce su tarjeta y confirma la operación (puede incluir verificación 3D Secure).
6
Redsys notifica al backend el resultado del pago
Redsys envía una notificación HTTP al backend (servidor a servidor) a la URL de notificación configurada.

// Redsys → Backend (notificación servidor-a-servidor)
POST /api/v1/pagos/notificacion/
{
  "Ds_SignatureVersion": "HMAC_SHA256_V1",
  "Ds_MerchantParameters": "...",
  "Ds_Signature": "..."
}
El backend verifica la firma de Redsys, decodifica los parámetros y comprueba el código de respuesta (Ds_Response). Si el pago es correcto (código 0000-0099), actualiza el estado del pedido a pendiente (pagado, pendiente de preparación). Si falla, marca el pedido como pago_fallido.
7
El usuario es redirigido de vuelta a la aplicación
Tras completar el pago, Redsys redirige al usuario a GET /pagos/resultado/, que a su vez redirige al frontend a la pantalla de confirmación con el código QR del pedido.
8
La cafetería gestiona el pedido
El personal de cafetería ve el nuevo pedido en el panel de administración (GET /pedidos/admin/). Actualiza el estado según avanza la preparación:

// Transiciones de estado del pedido:
pendiente_pago → pendiente → preparando → listo → entregado
                      ↓
                 pago_fallido

// Staff actualiza estado
PATCH /api/v1/pedidos/42/estado/
{ "estado": "preparando" }
9
El cliente recoge su pedido
El cliente muestra el código QR o alfanumérico (PB2401) en la cafetería. El personal verifica el código (GET /pedidos/verificar/PB2401/), entrega el pedido y actualiza el estado a entregado.

5. Gestión de Errores

5.1. Estructura estándar de respuesta de error

Todas las respuestas de error siguen una estructura uniforme para facilitar el manejo en el frontend:

{
  "error": {
    "codigo": "TOKEN_EXPIRADO",
    "mensaje": "El token de acceso ha expirado. Utiliza el refresh token para obtener uno nuevo.",
    "detalles": null
  }
}

En caso de errores de validación, el campo detalles contiene los errores específicos por campo:

{
  "error": {
    "codigo": "VALIDACION_FALLIDA",
    "mensaje": "Los datos enviados contienen errores.",
    "detalles": {
      "items": ["Este campo es obligatorio."],
      "franja_horaria_id": ["La franja horaria seleccionada ya no está disponible."]
    }
  }
}

401 Unauthorized Token expirado TOKEN_EXPIRADO "El token de acceso ha expirado."

403 Forbidden Permisos insuficientes (cliente intenta acceder a endpoints de dueño) PERMISOS_INSUFICIENTES "No tienes permisos para realizar esta acción."

500 Internal Server Error Error interno del servidor ERROR_INTERNO "Ha ocurrido un error interno. Inténtalo de nuevo más tarde."
503 Service Unavailable Servicio de Redsys no disponible PASARELA_NO_DISPONIBLE "La pasarela de pago no está disponible temporalmente."

5.3. Manejo de errores en el frontend

El frontend debe implementar la siguiente lógica para manejar los errores de forma consistente:

Código HTTPAcción del frontend
400Mostrar los mensajes de error junto a los campos del formulario correspondientes.
401 con código TOKEN_EXPIRADOIntentar renovar el token automáticamente con POST /auth/refresh/. Si falla, redirigir al login.
401 con otros códigosRedirigir al usuario a la pantalla de login.
403Mostrar mensaje "No tienes permisos" y redirigir a la pantalla principal.
404Mostrar pantalla de "No encontrado".
422 (pago)Mostrar mensaje de error de pago y ofrecer reintentar.
429Mostrar aviso de límite excedido con cuenta atrás.
500 / 503Mostrar pantalla genérica de error con opción de reintentar.
